## Context

当前 `seed-runner.ts` 仅支持幂等插入（按标题检查，存在则跳过）。当需要更新剧本内容（如补全完整版本）时，开发者无法通过 `pnpm seed` 完成更新，必须手动操作数据库。

**依赖关系：** 此 change 是三个剧本转换 change 的前置条件——转换完成后需要通过 `--force` 模式更新数据库。

## Goals / Non-Goals

**Goals:**
- 添加 `--force` 命令行参数支持更新已存在的剧本
- 更新时替换 content、重建 characters、创建新 version
- 保持默认行为不变（幂等跳过）

**Non-Goals:**
- 不支持部分更新（只支持整体替换）
- 不实现增量 diff 检测
- 不修改数据库 schema

## Decisions

### 1. 命令行参数设计

**决策：** 使用 `--force` 标志

**理由：**
- 语义清晰——强制覆盖
- 与 `--check` 参数风格一致
- 简单直观

**替代方案：**
- ❌ `--update` — 语义模糊（更新什么？）
- ❌ 环境变量控制 — 不够直观
- ❌ 默认总是更新 — 破坏幂等性

### 2. 更新策略

**决策：** 整体替换策略
1. 更新 `scripts` 表的 content 字段
2. 删除旧的 `characters` 记录，重新插入
3. 在 `script_versions` 创建新版本

**理由：**
- 简单可靠，避免增量合并的复杂性
- 角色列表可能变化（新增/删除），全量替换最安全
- 创建新版本保留历史，可回退

### 3. 版本号管理

**决策：** 查询当前最大版本号 +1

**理由：**
- 避免硬编码版本号
- 支持多次更新

## Risks / Trade-offs

**[风险 1] 角色ID变更**
- 风险：删除重建角色会导致角色 ID 变化，如果有外键引用可能出问题
- 缓解：检查 characters 表是否有被其他表引用的外键

**[风险 2] 版本膨胀**
- 风险：多次 `--force` 会创建大量版本
- 缓解：这是开发阶段可接受的行为

## Migration Plan

1. 修改 `seed-runner.ts` 添加 `--force` 参数解析
2. 实现 `updateScript()` 函数
3. 修改 `seedScript()` 函数在 `--force` 模式下调用更新逻辑
4. 测试：先正常 seed，再用 `--force` 验证更新
